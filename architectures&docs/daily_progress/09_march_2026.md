## Progress Update — 9 March 2026

### Work Completed Today

#### 1. Terraform Initial Setup

We initialized Terraform in the project and configured the AWS provider.

**File Added**

```
terraform/provider.tf
```

**Why**
This file defines:

* The Terraform version requirement
* The AWS provider configuration
* The AWS region where infrastructure will be deployed

This ensures consistent Terraform behavior across environments and allows Terraform to communicate with AWS services.

---

#### 2. Terraform Remote Backend (Bootstrap)

We created the bootstrap infrastructure required for Terraform state management.

**Files Added**

```
terraform/bootstrap/main.tf
terraform/backend.tf
```

**Resources Created**

* S3 bucket for Terraform state storage
* DynamoDB table for Terraform state locking

**Why**
Terraform state must be stored remotely to prevent loss of infrastructure information and allow safe collaboration.
The S3 bucket stores the Terraform state file, while the DynamoDB table prevents concurrent Terraform operations by locking the state during execution.

---

#### 3. VPC Module Creation

We started building a reusable Terraform module for networking infrastructure.

**Folder Added**

```
terraform/modules/vpc
```

**Files Added**

```
main.tf
variables.tf
outputs.tf
igw.tf
subnets.tf
route-tables.tf
```

**Why**
Infrastructure modules allow reusable and maintainable Terraform code.
The VPC module will be used by multiple environments such as dev and prod.

---

#### 4. Network Infrastructure Implemented

Inside the VPC module we implemented the core networking components.

**Resources Implemented**

* VPC creation
* Internet Gateway
* Public subnets across multiple availability zones
* Private subnets across multiple availability zones
* Public route table
* Route table associations

**Why**
This architecture creates a multi-AZ network design that supports high availability and proper internet routing for public services while keeping internal services isolated in private subnets.

---

#### 5. Project Structure Improvement

We organized Terraform resources into separate files inside the module.

**Why**
Splitting infrastructure resources into dedicated files improves readability and maintainability and aligns with professional Terraform project structures.
